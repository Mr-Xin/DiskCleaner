# DiskCleaner 项目规划文档

> macOS 磁盘管理工具 · 开源项目 · 纯 Swift（可少量 Objective-C / C）
>
> 文档版本：v1.0 · 编写日期：2026-05-26

---

## 1. 项目概述

DiskCleaner 是一款面向 macOS 的磁盘管理与清理工具，目标是帮助用户看清磁盘空间被什么占用、安全地回收无用空间，并干净地卸载应用。

四大核心功能（本次确定全部纳入范围）：

1. **磁盘空间可视化** —— 扫描磁盘并以图形方式展示各文件夹占用，类似 DaisyDisk / GrandPerspective。
2. **垃圾清理** —— 识别并清理系统/用户缓存、日志、临时文件、开发者垃圾等。
3. **大文件 / 重复文件查找** —— 找出超大文件和内容重复的文件，辅助手动清理。
4. **应用卸载** —— 卸载 App 时一并清除其散落在系统各处的残留文件。

定位为开源项目，因此分发方式、许可证、文档与社区协作也是规划的一部分。

### 设计原则

- **安全第一**：默认"移到废纸篓"而非永久删除；任何删除前都让用户确认；维护一份受保护路径黑名单。
- **可解释**：每一项被标记为"垃圾"的文件都要能说清它是什么、为什么可清理、清理后影响。
- **性能可接受**：要能在合理时间内扫描包含数百万文件的磁盘，且全程可取消。
- **核心逻辑与 UI 解耦**：扫描、哈希、规则、删除等逻辑独立成包，可独立测试。

## 3. 功能拆解

### 3.1 磁盘空间可视化

**用户故事**：选择一块磁盘或一个文件夹，几十秒内看到一张"空间地图"，一眼定位谁在吃空间，并能逐层下钻。

要做的事：

- 扫描指定卷 / 目录，构建一棵带尺寸的文件树（`FileNode`）。
- 计算每个目录的累计大小，支持按"实际占用磁盘空间"或"逻辑大小"两种口径。
- 以**矩形树图（Treemap）**为主视图渲染（v1）；环形图（Sunburst）作为可选项后置。
- 支持下钻 / 返回、悬停查看路径与大小、右键"在访达中显示"、直接选中删除。
- 顶部展示卷的总容量 / 已用 / 真实可用空间。

为什么 v1 选 Treemap 而非 Sunburst：矩形树图对空间利用率更高、信息密度更大、布局算法（squarified treemap）成熟，DaisyDisk 式的环形图更美观但实现和交互成本更高，留到后续迭代。

### 3.2 垃圾清理

**用户故事**：打开应用，看到按类别归集的可清理项与预估可回收空间，勾选后一键清理。

垃圾分类目录（初版规则集，每条规则都标注安全等级）：

| 类别 | 典型路径 / 内容 | 安全等级 |
|---|---|---|
| 用户缓存 | `~/Library/Caches/*` | 安全 |
| 应用日志 | `~/Library/Logs/*` | 安全 |
| 废纸篓 | `~/.Trash`、各卷 `.Trashes` | 安全 |
| 浏览器缓存 | Safari / Chrome / Firefox 缓存目录 | 安全 |
| 开发者垃圾 | Xcode `DerivedData`、`iOS DeviceSupport`、模拟器缓存、`Archives` | 安全 |
| 包管理器缓存 | npm / yarn / pip / CocoaPods / SwiftPM 缓存 | 安全 |
| 旧 iOS 设备备份 | `~/Library/Application Support/MobileSync/Backup` | 需确认 |
| 邮件附件下载 | `~/Library/Containers/com.apple.mail/.../Downloads` | 需确认 |
| 系统缓存 | `/Library/Caches/*`（需 root，v1 可暂不处理） | 需确认 |
| 大型旧下载 | `~/Downloads` 中超过 N 天未访问的大文件 | 需确认 |

设计要点：

- 规则用数据描述（建议 JSON 或 Swift 结构体声明），每条规则含：`id`、名称、类别、路径/通配符、安全等级、说明文案、大小估算方式。
- "安全"类可默认勾选；"需确认"类默认不勾选，必须用户手动选择。
- 绝不盲删应用的数据目录（`Application Support`、`Containers` 里的非缓存数据）。
- 清理前先估算大小并展示，清理后给出实际回收量。

### 3.3 大文件 / 重复文件查找

**大文件查找**：

- 在扫描结果基础上，按大小排序列出超过阈值（如 100 MB，可调）的文件。
- 支持按类型、修改时间、所在目录过滤。
- 列表项支持预览、在访达中显示、移到废纸篓。

**重复文件查找**（三级流水线，逐级收窄，避免对海量文件全量哈希）：

1. **按大小分组** —— 大小不同的文件一定不重复，先排除。
2. **局部哈希** —— 对每组内文件取首尾各若干 KB 做快速哈希，进一步分组。
3. **完整哈希** —— 对仍同组的文件做完整内容哈希（v1 用 CryptoKit 的 SHA-256，后续可换 xxHash/BLAKE3 提速），必要时再逐字节比对确认。

关键注意点：

- **硬链接**：同一 inode 的多个路径不是"重复"，删除不释放空间，需用 `fileResourceIdentifier` 去重。
- **APFS 克隆文件**：克隆文件内容相同但共享物理存储，删除"副本"同样不释放空间，需识别并标注。
- 每组重复文件中，由用户决定保留哪个，工具只做建议（如保留最早/路径最短的）。

### 3.4 应用卸载

**用户故事**：选中一个 App，工具列出该 App 本体及其散落在系统各处的关联文件，确认后一并清除。

实现路径：

- **枚举已安装应用**：扫描 `/Applications`、`~/Applications`，读取每个 `.app` 的 `Bundle Identifier` 与名称。
- **定位残留**：按 bundle id 与应用名，在以下位置搜索关联文件——
  - `~/Library/Application Support/`、`Caches/`、`Logs/`
  - `~/Library/Preferences/`（`com.vendor.app.plist`，可按 bundle id 精确匹配）
  - `~/Library/Containers/`、`Group Containers/`（沙盒应用，目录名即 bundle id，精确匹配）
  - `~/Library/Saved Application State/`、`WebKit/`、`HTTPStorages/`、`Cookies/`
  - `~/Library/LaunchAgents/`、`/Library/LaunchAgents/`、`/Library/LaunchDaemons/`、`PrivilegedHelperTools/`
- **匹配置信度**：bundle id 精确匹配为高置信度，应用名模糊匹配为低置信度，分组展示并默认只勾选高置信度项。
- **登录项 / 后台服务**：发现 LaunchAgent/Daemon 时，删除前先用 `launchctl` 卸载。
- 删除前完整列出待删文件清单，由用户确认。

可参考开源项目 **Pearcleaner**（SwiftUI 编写的开源卸载工具），其残留定位逻辑与 SwiftUI 结构都很有借鉴价值。

## 4. 技术选型

### 4.1 语言

纯 Swift 为主，性能敏感的目录遍历可通过一个薄薄的 Objective-C / C 桥接层调用 `getattrlistbulk`（见第 7 节）。这与你"纯 Swift，可混一点 OC"的选择一致。

### 4.2 UI 框架：建议从 AppKit/Storyboard 迁移到 SwiftUI

当前工程是 AppKit + Storyboard。对于一个 2026 年起步的新项目，建议：

- **应用外壳与绝大多数界面用 SwiftUI**：开发效率高、状态管理清晰、与 async/await 配合好。
- **自定义可视化（Treemap）用 SwiftUI `Canvas` 绘制**；若后期性能不够，再用 `NSViewRepresentable` 包裹 CoreGraphics/Metal 实现。

两种迁移方式：

- 方案 A（推荐）：新建 SwiftUI App 生命周期的工程结构，弃用 `Main.storyboard`。新项目这样做最干净。
- 方案 B：保留 AppKit `AppDelegate`，用 `NSHostingController` 承载 SwiftUI 视图。适合想保留部分 AppKit 代码时。

> 决策建议：选方案 A。当前工程几乎是空的，迁移成本极低，越早决定越省事。

### 4.3 并发

采用 Swift 结构化并发：`async/await` + `actor`（保护扫描过程中的共享累加状态）+ `TaskGroup`（并行遍历子目录）+ `AsyncStream`（向 UI 流式推送扫描进度）。全流程可 `cancel`。

### 4.4 最低系统版本

建议部署目标定为 **macOS 13 Ventura（最低）/ 14 Sonoma（推荐）**。Ventura 起 `SMAppService` 等 API 可用，SwiftUI 也足够成熟。具体取舍取决于你想用的 SwiftUI 新特性——定下来后写进工程配置并在 CI 固定。

## 5. 权限与分发模型（关键约束，务必先想清楚）

这是磁盘清理类工具最容易踩坑、且会反向影响架构的地方。

### 5.1 沙盒 vs 完全磁盘访问

- 一个真正可用的磁盘清理工具需要读取 `~/Library`、其他用户目录、邮件、各应用数据等位置，这些都受 **TCC（完全磁盘访问，Full Disk Access）** 保护。
- **App Sandbox 沙盒**会严格限制文件访问，沙盒应用拿不到完全磁盘访问，只能通过"用户主动选择文件夹 + security-scoped bookmark"访问有限范围——这会让清理功能严重残废。
- **完全磁盘访问无法通过 API 申请**，只能引导用户去"系统设置 → 隐私与安全性 → 完全磁盘访问"手动开启。

**结论**：DiskCleaner 应**以非沙盒方式发布**，并引导用户授予完全磁盘访问。

### 5.2 由此带来的连锁影响

- **不能上架 Mac App Store**（App Store 要求开启沙盒）。这与"开源项目"的定位并不冲突——通过下列渠道分发即可：
  - GitHub Releases 提供 `.dmg`
  - Homebrew Cask
- **代码签名与公证**：在 App Store 外分发，需要用 Apple Developer Program 账号（99 美元/年）对应用做签名 + 公证（notarization），否则用户首次打开会遇到 Gatekeeper 拦截。开发early阶段可用自签名，正式发布前务必准备好付费开发者账号。
- **权限自检**：应用启动时尝试读取一个受 TCC 保护的已知路径（如 `~/Library/Mail`），读取失败即说明未授予完全磁盘访问，弹出引导界面。

### 5.3 是否需要特权助手

清理用户自己的文件不需要 root 权限。只有清理系统级目录（如 `/Library/Caches`）才需要。

**建议**：v1 不引入特权助手，遇到需要 root 的项目直接跳过或尝试后报告失败即可。若 v2 确实要做系统级清理，再用 `SMAppService` 注册特权助手工具。

## 6. 架构设计

### 6.1 分层

```
┌─────────────────────────────────────────┐
│  App 层 (SwiftUI)                         │
│  Views + ViewModels (MVVM)               │
├─────────────────────────────────────────┤
│  DiskCleanerCore (Swift Package, 无 UI)   │
│  Scanner / HashService / DuplicateFinder │
│  JunkRulesEngine / AppUninstaller        │
│  DeletionService / ProtectedPaths        │
│  PermissionsChecker                      │
└─────────────────────────────────────────┘
```

**为什么把核心逻辑抽成独立的 Swift Package**：

- 可测试 —— 纯逻辑不依赖 UI，能用单元测试覆盖。
- 边界清晰 —— UI 与逻辑解耦，便于并行开发与重构。
- 可复用 —— 将来若想做命令行版本（CLI），直接复用 Core 即可。

### 6.2 核心组件职责

- **Scanner**：遍历文件系统，产出 `FileNode` 树，流式上报进度，可取消。
- **FileNode**：文件/目录的模型——路径、名称、是否目录、逻辑大小、磁盘占用、inode 标识、修改时间等。
- **HashService**：流式计算文件哈希（局部 + 完整）。
- **DuplicateFinder**：三级流水线找重复文件，处理硬链接与 APFS 克隆。
- **JunkRulesEngine**：加载规则集，匹配垃圾文件，估算可回收空间。
- **AppUninstaller**：枚举应用、定位残留、计算匹配置信度。
- **DeletionService**：执行删除（默认移废纸篓），写审计日志。
- **ProtectedPaths**：受保护路径黑名单，删除前强制校验。
- **PermissionsChecker**：检测完全磁盘访问是否已授予。

### 6.3 数据流

UI 发起扫描 → Scanner 在后台并发遍历 → 通过 `AsyncStream` 把进度与节点流式推给 ViewModel → ViewModel 更新 `@Published` 状态 → SwiftUI 视图刷新。删除操作走 DeletionService，先过 ProtectedPaths 校验，再移入废纸篓并记审计日志。

## 7. 关键技术难点与对策

### 7.1 大规模扫描的性能

- `FileManager.enumerator` 配合预取资源键（`URLResourceKey`）对中小目录够用，但面对数百万文件会偏慢。
- **最大的性能杠杆是 `getattrlistbulk`**（C API）：一次系统调用批量取回多个文件的属性，比逐文件 `stat` 快一个数量级。通过一个薄 OC/C 桥接层调用它。
- 配合 `TaskGroup` 并行遍历不同子树。
- 建议先用 `FileManager` 实现一个能跑的版本，再用 `getattrlistbulk` 做性能优化——不要一开始就陷在底层 API 里。
- 需要的资源键：`isDirectoryKey`、`totalFileAllocatedSizeKey`（实际磁盘占用）、`fileSizeKey`（逻辑大小）、`fileResourceIdentifierKey`（inode/硬链接去重）、`isSymbolicLinkKey`、`isPackageKey`（把 `.app` 当整体）、`contentModificationDateKey`、`volumeIdentifierKey`。

### 7.2 APFS 带来的容量计算陷阱

- **克隆文件**共享物理块——多个文件逻辑大小之和会高估实际占用。用 `totalFileAllocatedSize` 而非 `fileSize`。
- **快照与可清除空间（purgeable）**——磁盘"已用"里有一部分是系统可随时回收的。展示"真实可用空间"用 `volumeAvailableCapacityForImportantUsageKey`。
- **硬链接**——按 inode 去重，避免重复计数。
- 在 UI 上把数字明确标注为"估算值"，不要给用户精确到字节的虚假承诺。

### 7.3 扫描时应跳过的位置

- `/System`（SIP 保护、只读卷）、`/private/var/db` 等系统目录。
- 其他卷的挂载点（除非用户主动选择）。
- firmlink 造成的路径重复。
- `.photoslibrary` 等包内部结构（当作整体）、Time Machine 本地快照。
- 不跟随符号链接，避免环路与重复计数。

### 7.4 重复文件检测的正确性

见 3.3：三级流水线 + 硬链接/克隆识别。这是"找重复"功能可信度的核心，务必用构造好的测试夹具覆盖。

### 7.5 应用残留定位的准确性

bundle id 精确匹配可靠；应用名模糊匹配容易误伤（例如名字含通用词的 App）。务必区分置信度、默认只勾高置信度、删除前让用户逐项确认。

## 8. 安全设计（防误删，重中之重）

误删用户数据是这类工具最严重的事故，必须从设计上系统性防范：

- **默认移到废纸篓**：用 `FileManager.trashItem(at:resultingItemURL:)`，可恢复。永久删除作为可选项，藏在额外确认之后。
- **受保护路径黑名单**：硬编码一份绝不允许删除的路径集合——根目录、`/System`、`/usr`（非 `/usr/local`）、`/bin`、`/sbin`、用户主目录根、`~/Library/Keychains`、TCC 数据库目录等。任何删除操作先过这道校验。
- **预演模式（Dry Run）**：先列出"将要删除什么、各多大"，用户确认后才真正执行。
- **多级确认**：批量删除、永久删除、清理"需确认"类目时，都要显式二次确认。
- **审计日志**：记录每次删除了什么、多大、何时，供用户事后查看。
- **保守的默认规则**：拿不准的规则一律标"需确认"且默认不勾选。
- **首次运行免责声明**：明确告知用户工具会删除文件，使用风险自负（开源项目尤其要写清楚）。

## 9. 开发里程碑

时间为相对工作量参考，非承诺日期。建议每个阶段结束都打一个可运行的 tag。

### Phase 0 — 项目奠基

- 决策并落地 UI 框架（建议迁移到 SwiftUI App，弃用 Storyboard）。
- 建立 `DiskCleanerCore` Swift Package，划好模块骨架。
- 加入 `LICENSE`、`README`、`.gitignore`。
- 配置 GitHub Actions CI：在 macOS runner 上跑 build + test。

### Phase 1 — 磁盘空间可视化（MVP）

- `PermissionsChecker` + 完全磁盘访问引导界面。
- `Scanner`（先 `FileManager` 版，再 `getattrlistbulk` 优化）+ `FileNode` 树。
- Treemap 视图、下钻交互、卷容量展示。
- **交付**：能选定一块磁盘/文件夹，看清空间被谁占用。

### Phase 2 — 大文件 / 重复文件查找

- 大文件列表（排序、过滤）。
- `DuplicateFinder` 三级流水线 + 硬链接/克隆处理。
- 结果界面 + 移废纸篓。

### Phase 3 — 垃圾清理

- `JunkRulesEngine` + 初版规则目录。
- 分类清理界面，区分"安全/需确认"，展示预估与实际回收量。

### Phase 4 — 应用卸载

- 应用列表、按 bundle id 定位残留、置信度分组。
- LaunchAgent/Daemon 卸载、确认界面、执行删除。

### Phase 5 — 打磨与发布

- 审计日志界面、从废纸篓恢复的引导、设置项、中英文本地化。
- 代码签名 + 公证、打包 `.dmg`、接入 Sparkle 实现自动更新。
- 完善文档，发布 v1.0。

- **版本与分发**：语义化版本号；Sparkle 负责应用内自动更新。

| 风险 | 影响 | 应对 |

| 误删用户数据 | 严重，可能损失文件 | 默认移废纸篓、受保护路径黑名单、预演模式、多级确认、审计日志、保守默认规则 |
| 容量数字不准 | 用户不信任工具 | 用 `totalFileAllocatedSize`、按 inode 去重、识别 APFS 克隆、数字标注为"估算" |
| 完全磁盘访问引导摩擦 | 用户流失在第一步 | 清晰的 onboarding 流程，自动检测授权状态并复检 |
| 海量文件扫描慢 | 体验差 | `getattrlistbulk`、`TaskGroup` 并行、增量刷新 UI、可取消 |
| macOS 版本 / TCC 策略变化 | 升级后功能失效 | 固定部署目标、多系统版本测试、隔离 OS 相关路径逻辑 |
| 无法上架 App Store | 分发渠道受限 | 已接受——走 DMG / Homebrew，开源定位本就如此 |
| 签名与公证成本 | 用户遇 Gatekeeper 警告 | 准备 Apple Developer 付费账号，发布前完成公证 |
| 法律 / 免责 | 删除文件引发纠纷 | README 与首次运行明确免责声明，风险自负 |
| 开源维护负担 | issue 堆积、范围蔓延 | 控制功能范围、用模板规范 issue、明确 roadmap |

## 12. 参考项目与资料

**可借鉴的开源 / 同类工具**：

- **OmniDiskSweeper** —— 极简列表式磁盘分析参考。

**关键 API / 文档方向**：

- 文件遍历：`FileManager`、`URLResourceValues` / `URLResourceKey`、`getattrlistbulk`(2) man page。
- 哈希：`CryptoKit`（SHA-256）；提速可选 xxHash / BLAKE3 的 SwiftPM 包。
- 删除：`FileManager.trashItem`。
- 权限与分发：App Sandbox、Full Disk Access (TCC)、"Notarizing macOS software" 官方文档。
- 特权助手（如 v2 需要）：`ServiceManagement` / `SMAppService`。
- 自动更新：Sparkle。
